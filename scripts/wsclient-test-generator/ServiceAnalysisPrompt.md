# AI Prompt: Service-Level Discovery & Dependency Analysis

## 🎯 Objective

You are an AI agent responsible for performing a **service-level analysis** of an HPCC Java client class. Your goal is to discover all business methods, analyze inter-method dependencies, and produce two outputs:

1. **A Markdown analysis file** (human-readable, used as context in subsequent per-method analysis steps)
2. **A JSON method list file** (machine-readable, consumed directly by the Python orchestrator)

## 🧩 System Context

### Client-Side (Java)
- The client library uses Apache Axis2 to generate request and response objects from WSDL definitions
- These objects are wrapped by autogenerated wrapper classes to avoid direct dependency on Axis2 classes
- Each server-side API group is represented by a client class of the form: `HPCC${ServiceName}Client`

### Server-Side (C++)
- Service implementations are defined in: `esp/services/ws_${ServiceName}`
- Request and response object definitions (IDL) are in: `esp/scm/ws_${ServiceName}.ecm`

## 🔍 File Search Strategy

> ⚠️ **Important**: File and directory names may differ in capitalisation or have minor naming differences from what is expected. **Always use fuzzy/case-insensitive searches** — never assume an exact path is correct.
>
> ⚠️ **Always search from the project root directories provided in the context above** (HPCC4J Project Root and HPCC Platform Source Root). Do NOT use `.` as a search root — the script may be run from a different directory and `.` will not be the project root.

**Required approach when locating any file:**

1. **Use `find -iname` from the project root** (case-insensitive), not from `.`:
   ```bash
   # Java client class — search from HPCC4J project root
   find <HPCC4J_PROJECT_ROOT>/wsclient/src -iname "*${ServiceName}Client*" -type f
   find <HPCC4J_PROJECT_ROOT>/wsclient/src -iname "HPCC*${ServiceName}*" -type f

   # HPCC Platform source files — search from HPCC Platform source root
   find <HPCC_PLATFORM_SOURCE_ROOT>/esp/services -maxdepth 1 -iname "*${ServiceName}*" -type d
   find <HPCC_PLATFORM_SOURCE_ROOT>/esp/scm -iname "*.ecm" | xargs grep -il "${ServiceName}"
   ```

2. **Try multiple name variants** if the first search returns nothing:
   - With and without the `Ws`/`WS` prefix (e.g., `WsStore` → `Store`, `store`)
   - All-lowercase, PascalCase, camelCase variants
   - Partial name globs (e.g., `*store*client*`, `*Store*Client*`)
   - Underscore vs. no-separator variants (e.g., `ws_store` vs `wsstore`)

3. **Use `grep -ril` from the project root** when a name-based search fails:
   ```bash
   grep -ril "class HPCC.*${ServiceName}Client" <HPCC4J_PROJECT_ROOT>/wsclient/src
   grep -ril "${ServiceName}" <HPCC_PLATFORM_SOURCE_ROOT>/esp --include="*.java"
   ```

4. **For ESP source paths**, note that the directory may be named with a lower-case stripped variant of the service name (e.g., `ws_store` for `WsStore`, `ws_dfu` for `WsDFU`). Try:
   ```bash
   find <HPCC_PLATFORM_SOURCE_ROOT>/esp/services -maxdepth 1 -iname "*store*" -type d
   find <HPCC_PLATFORM_SOURCE_ROOT>/esp/scm -iname "*.ecm" | xargs grep -il "${ServiceName}"
   ```

5. **Never give up after one failed search** — try at least 3 different variants/strategies before treating a file as missing.

## 🧠 Task Instructions

### Step 1 — Discover Business Methods

Read `HPCC${ServiceName}Client.java` and identify all **business methods**:

**Include:**
- All `public` non-static methods that perform service operations
- Methods that send requests to the HPCC server
- Methods that wrap or delegate to other business methods

**Exclude:**
- Factory/singleton methods (e.g., `get(...)`, `get(Connection)`)
- Infrastructure methods (e.g., `getServiceURI()`, `getDefaultStub()`, `getServiceWSDLURL()`, `getServiceWSDLPort()`)
- Constructors
- Methods inherited from `BaseHPCCWsClient` that are not overridden
- `protected` or `private` methods (unless they are called by public business methods — note those as internal dependencies)

### Step 2 — Analyze Each Method

For each discovered business method, record:

| Field | Description |
|-------|-------------|
| **Name** | Method name |
| **Signature** | Full method signature (return type, name, parameters) |
| **Overloads** | Whether this method has overloaded variants |
| **Group** | Functional group (e.g., Health, Store Management, KV CRUD, Encrypted Operations, Listing/Query) |
| **Calls Other Methods** | Other business methods this method calls internally (e.g., `fetchValueEncrypted` calls `fetchValue`) |
| **Server State Effect** | What server-side state this method creates, reads, updates, or deletes (CRUD classification) |
| **Prerequisites** | What state must exist for this method to succeed (e.g., `fetchValue` requires a prior `setValue`) |

### Step 3 — Build Dependency Graph

Analyze dependencies between methods:

1. **Data dependencies:** Method A must create state that method B reads (e.g., `createStore` → `setValue` → `fetchValue`)
2. **Internal call dependencies:** Method A delegates to method B internally (e.g., `fetchValueEncrypted` wraps `fetchValue`)
3. **Cleanup dependencies:** Method A removes state that method B creates (e.g., `deleteNamespace` removes what `setValue` created)

### Step 4 — Determine Analysis Order

Compute a **topological ordering** of methods based on dependencies. Methods that have no dependencies on other methods come first. Methods that depend on understanding other methods come later.

The goal: when each method is analyzed in this order during per-method analysis (Step 1 of the main pipeline), the AI already understands the prerequisite methods.

### Step 5 — Classify Functional Groups

Group methods by their functional area. Examples:
- **Health/Connectivity:** `ping`
- **Store Management:** `createStore`, `listStores`
- **Key-Value CRUD:** `setValue`, `fetchValue`, `deleteValue`, `listNSKeys`, `fetchAllNSKeys`, `fetchKeyMetaData`
- **Encrypted Operations:** `setValueEncrypted`, `fetchValueEncrypted`
- **Namespace Management:** `listNamespaces`, `deleteNamespace`

Adjust groups based on the actual methods discovered — these are examples from WsStore.

### Step 6 — Test Independence Guidelines

Provide guidelines for writing independent tests:

- Each test creates a uniquely-named store/namespace (e.g., `"testStore_" + methodName + "_" + UUID.randomUUID().toString().substring(0, 8)`)
- Each test performs its own setup (e.g., a `fetchValue` test calls `setValue` first within the test body)
- Each test cleans up after itself in a `finally` block
- Tests must NOT depend on `@FixMethodOrder` execution ordering for correctness
- Shared helper methods (e.g., `createTestStore()`, `cleanupStore()`) should be extracted into the test class

## 📄 Output 1: Markdown Analysis File

Generate a Markdown file with the following sections:

### A. Method Inventory Table

| # | Method Name | Signature | Group | Calls Other Methods | Server State Effect | Prerequisites |
|---|-------------|-----------|-------|---------------------|---------------------|---------------|
| 1 | ... | ... | ... | ... | ... | ... |

### B. Dependency Graph

A textual representation of how methods relate to each other.

### C. Functional Groups

List each group with its member methods.

### D. Test Independence Guidelines

Specific guidelines for writing independent tests for this service.

### E. Recommended Analysis Order

A numbered list of methods in topological/dependency order.

## 📄 Output 2: JSON Method List File

**CRITICAL**: In addition to the markdown file, you MUST also create a JSON file at: `${METHOD_LIST_FILE}`

This JSON file is consumed directly by the Python orchestrator. It must conform to this exact schema:

```json
{
  "service": "${ServiceName}",
  "clientClass": "HPCC${ServiceName}Client",
  "methods": [
    {
      "name": "<methodName>",
      "signature": "<full method signature>",
      "group": "<functional group>",
      "callsOtherMethods": ["<method1>", "<method2>"],
      "serverStateEffect": "<Create|Read|Update|Delete|None>",
      "analysisOrder": <integer starting from 1>
    }
  ]
}
```

**Requirements:**
- The `methods` array MUST be sorted by `analysisOrder`
- `name` values MUST exactly match the Java method names from the source code
- If a method has overloads, include one entry using the base method name (the overloads will be handled during per-method analysis)
- `analysisOrder` is the topological ordering from Step 4
- `callsOtherMethods` should only reference other methods in the `methods` array
- `serverStateEffect` should be one of: `Create`, `Read`, `Update`, `Delete`, `Create/Update`, `Read/Delete`, or `None`

**Use the `create_file` tool to save this JSON file to: `${METHOD_LIST_FILE}`**
